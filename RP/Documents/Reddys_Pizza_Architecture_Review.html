1. Executive Summary

The Reddy's Pizza Platform is a cloud-native, microservices-based, and highly scalable ordering and management ecosystem built on Microsoft Azure. Designed to meet the demands of a high-throughput, low-latency Quick-Service Restaurant (QSR) environment, the platform caters to three primary stakeholder groups:

  • End Customers: Initiating orders via a cross-platform React Native mobile application. Supports rapid menu browsing, favorites management, and one-click reordering.
  • Operations & Admin Staff: Managing menus, deals, loyalty programs, and global business parameters through an administrative web dashboard.
  • inHouse Staff: Handling telephonic orders, walk-in customers, and store-specific workflows Shift management (Punch in / Punch out) using a specialized, lightning-fast web terminal.

To optimize the balance between strict microservices decoupling and operational simplicity, the architecture is designed from scratch with seven core microservices planned. These services execute containerized workloads inside Azure Kubernetes Service (AKS), leveraging Azure Service Bus for resilient, asynchronous, event-driven communication. This architectural pattern guarantees exceptional availability, insulates customer-facing ordering pipelines from downstream system delays, and provides high tolerance against external service outages.

3. Architecture Principles & Key Decisions

3.1 Guiding Principles

  • Customer experience first: Optimized response times for ordering, caching catalog metadata, and rendering customized user interfaces instantly.
  • Defense in depth: Security enforcement at edge (WAF), API gateway (JWT, TLS), and internal network (Virtual Network (VNet) private endpoints).
  • Passwordless authentication: Managed Identity for all service-to-service communication. Azure SQL and Blob Storage deny credentials via Service account & Principle.
  • Async by default: Service Bus topics decouple downstream processing (POS adapter, PDF invoices, Bulkgate messages) from client calls.
  • Strict Service Database Ownership: Independent databases for each service, hosted on a single server, physically enforcing data isolation and preventing cross-service data coupling.
  • Observable everything: Log Analytics workspace acts as a central telemetry store, enabling trace auditing through correlation IDs.

3.2 Key Architectural Decisions

Decision Area Choice Rationale
Service Count 7 Services (Designed from Scratch) Balances microservices separation and operational simplicity. Fits team capacity.
Database Strategy Database-per-Service on a Single Server Enforces strict service boundaries and independent schemas, avoiding cross-database joins.
Communication Async via Service Bus Isolates slow third-party services (SMS, PDF, POS, EMAIL) from customer API paths.
Auth Strategy Centralized JWT validation at APIM Decentralized services skip auth overhead. APIM validates and appends headers.
Frontend Hosting Azure Static Web Apps Zero-server hosting, built-in SSL and CDN, and automatic branch previews.
Secrets Strategy Key Vault + Workload Identity Allows keys rotation without code changes. Eliminates secrets from configurations.
AKS Orchestration AKS with NGINX Ingress Provides high-density pod hosting, rolling upgrades, and auto-scaling.

4.1 Description of the 7-Layer Infrastructure

  • Layer 1: Global Edge: Azure Front Door terminates SSL globally and applies WAF policies to defend against SQL injection (SQLi), Cross-Site Scripting (XSS), and DDoS attacks.
  • Layer 2: Regional VNet Ingress: Application Gateway acts as regional load balancer routing to internal API Management (APIM), which validates JWT tokens and rate-limits users.
  • Layer 3: Kubernetes Cluster Edge: NGINX Ingress controller routes external gateway requests to Kubernetes services inside the AKS private virtual network.
  • Layer 4: Bounded Context Microservices: Containerized .NET 10 API microservices representing Bounded Contexts (Auth, Store, Catalog, Order, Notification, Reporting, Payment).
  • Layer 5: Asynchronous Event Bus: Azure Service Bus topic-based publish-subscribe network that propagates state changes (e.g., OrderPlaced, PaymentCompleted) to downstream workers.
  • Layer 6: Relational & Cache Data Stores: Azure SQL Database Server (independent databases per microservice), Azure Cache for Redis (low-latency session/cart store), and Azure Blob Storage (media and invoice storage).
  • Layer 7: External Gateways: Payment processors (Stripe), messaging systems (Bulkgate, WhatsApp, FCM), and localized POS terminal systems.

6. Microservices Breakdown (7 Services)

The platform consists of 7 core microservices, each owning a specific domain and bounded context. All services run as containerized .NET 10 applications on AKS with 2 Pod replicas each (auto-scalable up to 10 Pods under load).

6.1 Auth Service

Owns: User authentication, roles, RBAC, addresses, profile management.

Responsibilities: JWT token generation and refresh; user registration and login (OTP, Google OAuth, email/password); role-based access control across customer, admin, store_manager, and store_staff roles; delivery address management with geocoding.

Tables owned: Users, Roles, Addresses.

Key technologies: JWT (RS256 signing), custom identity for customers, optional Entra ID integration for workforce SSO.

6.2 Store Service

Owns: Store configurations, operating boundaries, employee attendance, and localized operations.

Responsibilities: Store location coordinates; operating hours; settings; localized deals configurations; tracking employee attendance punch records.

Tables owned (DbSet mapping):

  • Businesses: Global business details.
  • BusinessDocs: Regulatory documents (e.g. FSSAI, GST certificates).
  • Stores: Individual branch details.
  • StoreHours: Branch operational hour ranges.
  • StoreProducts: Availability logs per store.
  • StoreSettings: Branch-level features toggles.
  • StoreDeals: Store-specific campaigns and promotions.
  • EmployeeAttendances: Staff shift punch-in and punch-out records.
  • StoreOrders & StoreOrderItems: Fulfillment dockets processed at store POS level.

6.3 Catalog Service

Owns: The global menu structure, items categorization, customization modifiers, and pricing matrices.

Responsibilities: Creating and updating menu structures; sizing rules; toppings details; and discount offers.

Tables owned (DbSet mapping):

  • Departments & SubDepartments: Menu sections hierarchy.
  • Crusts & Toppings: Custom modifiers options.
  • AddonCategories & Addons: Sides and customizations.
  • MenuItems, MenuItemCrusts, MenuItemSizes, MenuItemToppings, MenuItemAddons: Specific items and available selections.
  • Sizes: Standard size labels.
  • Combos & ComboMenuItems: Multi-item bundles.
  • Offers & StoreOfferOverrides: Marketing campaign rules and overrides.

6.4 Order Service

Owns: Carts management, order checkout, order history logs, and store availability caching.

Responsibilities: Shopping cart calculations; order placement validations; order state progressions; customer feedback ingestion.

Tables owned (DbSet mapping):

  • Carts & CartItems: Transient order builders.
  • Orders & OrderItems: Relational transaction records.
  • OrderStatusHistories: Order progression tracking.
  • Feedbacks: Post-fulfillment user reviews.
  • StoreHours & StoreSettings: Read-only DB views (synonyms) referencing Store Service databases to validate timing during order placement.

6.5 Payment Service

Owns: Gateway interactions, transaction logs, and invoicing state.

Responsibilities: Processing online cards via Stripe; validating payment notifications; managing cash-on-delivery state records.

Tables owned: Payments, PaymentTransactions.

Key technologies: Stripe SDK integrations, distributed transactional logs.

6.6 Notification Service

Owns: Outbound communications (SMS, email, push notifications, WhatsApp).

Responsibilities: Listening to event bus; selecting multi-channel templates; dispatching messages via Bulkgate and FCM.

Tables owned: Notifications (sent logs).

6.7 Reporting Service

Owns: Data aggregation, BI, and invoice generation.

Responsibilities: Querying microservice databases via read-replicas; building sales PDFs; delivering BI charts to administrators.

Tables accessed: Reads from all service databases; writes minimal analytics metadata.

7. Layered Security Architecture

The platform implements defense-in-depth with three distinct security layers, each providing independent protection against different attack vectors.

7.1 Layer 1: Azure Front Door + WAF (Global Edge)

Azure Front Door acts as the global edge router, routing traffic to the closest healthy Azure gateway. It enforces SSL 1.2/1.3 and applies OWASP Core Rule Set (CRS) policies to block common web attacks at the network perimeter. DDoS protection prevents application-level overload.

7.2 Layer 2: APIM + Application Gateway (VNet)

The regional Application Gateway (WAF-enabled) intercepts traffic from Front Door and routes it to the API Management (APIM) gateway, which is virtual network injected. APIM provides centralized JWT Verification, client rate limiting, and private IP network boundaries.

7.3 Layer 3: NGINX Ingress (In-Cluster)

NGINX Ingress Controller running inside the AKS cluster routes requests to appropriate microservices. It balances internal load across 2 Pod replicas, performs TCP/HTTP checks, and acts as internal service discovery.

7.4 Identity & Secrets

  • Workforce identity: Optional specific inHouse Microsoft Organization Account SSO integration for employee punch/attendance via Microsoft Entra ID.
  • Workload identity: Each AKS Pod has a Managed Identity (service principal in Entra). Used for Key Vault, SQL, Blob, and ACR access — no passwords or connection strings in pods.
  • Customer identity: Custom Identity service stores users in SQL, issues JWTs, handles OTP and Google OAuth flows.
  • Secret management: Azure Key Vault stores all connection strings, API keys, certificates, and signing keys.

8. Data Layer Architecture

8.1 Azure SQL Database (Primary Store)

Strategy: A single logical Azure SQL Database Server hosting 7 independent databases (one per microservice) to enforce physical data isolation. This database-per-service strategy prevents tight coupling of data structures, isolates processing resources, and matches bounded contexts.

Authentication: Managed Identity — no username or password. Connection string fetched from Key Vault at startup.

Service ownership: Each service has exclusive access to its database. Cross-database queries are avoided; services communicate via asynchronous events or REST APIs.

Indexes: Critical indexes placed on frequently queried foreign keys, search fields, and composite query filter columns (such as store identifiers and date ranges) to optimize lookup performance.

8.2 Azure Cache for Redis

  • Receipts and dockets: receipts/orders/{id}/receipt.pdf — generated by Reporting Service after order placement.

9. Platform Services

  • Azure Key Vault: Centralized encryption key and secret manager. Audited logs are fed directly to Log Analytics.
  • Microsoft Entra ID: Handles workload identity security and optional workforce SSO for inHouse systems.
  • Application Insights: Monitors telemetry, latency, errors, dependencies (SQL execution, HTTP calls), and user behaviors.
  • Log Analytics: Consolidates logs from AKS, APIM, SQL, Key Vault, and Front Door for cross-component queries.
  • Container Registry (ACR): Private container registry with integrated vulnerability scanner blocking vulnerable builds.

10. Async Event Processing (Service Bus)

10.1 Why Async Messaging?

Without Service Bus, the Order Service would synchronously call Notification, Reporting, and POS — leading to high latency and cascading failure risks. With Service Bus, the order is committed to the database and publishes the event to Service Bus, immediately returning a successful response to the customer while background workers process the event independently.

10.2 Topic & Subscription Topology

We utilize the standard Service Bus tier (sb-reddyspizza):

  • Topic: order-events: Single topic where the Order service publishes events.
  • Subscription: notification-sub: Consumed by Notification service to send SMS, WhatsApp, and Firebase push alerts.
  • Subscription: reporting-sub: Consumed by Reporting service to build invoice PDFs.
  • Subscription: pos-sub: Consumed by Store POS adapters to print kitchen dockets.

10.3 Reliability Features

  • Message Deduplication: Matches message IDs against order IDs, discarding repeats if client retries.
  • Retry & Backoff: Retries processing up to 10 times with exponential delay before sending to Dead-Letter Queue (DLQ).
  • Fault Isolation: If Bulkgate is down, only notifications are delayed; order execution and printing function normally.

12.2 Database Additions for Favourites and Reordering

To support Favourites and trace Reorder statistics, the following schema expansions are implemented in the database:

Table: ordering.Favorites

  • Id (PK, UNIQUEIDENTIFIER): Primary key of the favorite record.
  • UserId (FK, UNIQUEIDENTIFIER): References identity.Users.Id. Indexed for lookup.
  • ProductId (FK, UNIQUEIDENTIFIER): References catalog.Products.Id.
  • CustomizationJson (NVARCHAR(MAX)): JSON string describing crust size, toppings, and add-ons.
  • Alias (NVARCHAR(100)): Custom name provided by the user (e.g., 'My Cheesy Treat').
  • CreatedAt (DATETIME2): Timestamp representing when the record was saved.

Table: ordering.Orders (Schema Alterations)

  • ReorderedFromId (FK, UNIQUEIDENTIFIER, Nullable): A self-referencing foreign key linking to ordering.Orders.Id.
  • Channel (NVARCHAR(50)): Represents the order entry source (e.g., 'Mobile', 'Web_Admin', 'POS', 'Reorder_Mobile').

14. Deployment Architecture

14.1 Container Image Pipeline

  1. Developer pushes code to GitHub main branch.
  2. GitHub Actions / Azure DevOps pipeline triggers automatically.
  3. Docker image built (order-service: v1.2.4 tag from semantic version + git SHA).
  4. Image scanned for CVEs by Microsoft Defender for Containers.
  5. Image pushed to ACR (acrreddyspizza.azurecr.io).
  6. kubectl apply deploys new version to AKS with rolling update strategy.
  7. Health checks verify Pods are ready before routing traffic.

14.2 Kubernetes Topology

  • Namespace: default
  • Deployments: one per service (7 total), each with 2 Pod replicas
  • Resource limits: 100m CPU request, 500m limit; 128Mi memory request, 512Mi limit
  • Liveness probe: GET /health every 30s
  • Readiness probe: GET /ready every 10s
  • HPA auto-scaling: min 2 Pods, max 10 Pods, target 70% CPU
  • ServiceAccount with Workload Identity annotation links each Pod to its Managed Identity

16. Architecture Decision Records (ADRs)

ADR-001: Database-per-Service on a Single Server

Decision: Independent databases for each microservice, hosted on a single Azure SQL Database Server.

Context: Microservices best practice dictates database-per-service to prevent tight coupling of data structures. At the same time, maintaining separate servers for each service creates massive cost and administrative overhead.

Trade-offs: Enforces strict data boundary isolation (zero cross-database joins, only API-level integrations) while maintaining a centralized server for easy administration and cost optimization.

ADR-002: Sync vs Async Event Processing

Decision: Async via Azure Service Bus.

Context: Order placement triggers multiple downstream actions: SMS, push notification, PDF generation, POS docket printing. Synchronous calls would create cascading failure risks.

Trade-offs: Adds complexity, but enables optimized response times, isolated failures, and automatic retries.

ADR-003: APIM for JWT Validation

Decision: JWT validated at APIM layer, not in each service.

Context: Each service could validate JWTs independently, but this requires duplicate code, key distribution, and re-parsing on every request.

Trade-offs: Centralized validation simplifies services, reduces duplicate code, and enables centralized rate limiting and policy management.

ADR-004: 7 Microservices Architecture Bounded Contexts

Decision: Implement exactly 7 core microservices designed from scratch.

Context: Designing the system from scratch requires establishing clear domain boundaries. Splitting or merging services later incurs high migration costs, while over-fragmentation leads to operational complexity.

Trade-offs: Provides clean separation of concerns, manageable resource scaling (2 pods per service), and maps directly to the database-per-service topology, avoiding over-fragmentation.

ADR-005: Static Web Apps for Frontends

Decision: Azure Static Web Apps host React admin and inHouse POS apps.

Context: Static Web Apps are purpose-built for SPAs with built-in CI/CD, custom domain support, and authentication integration.

Trade-offs: Limited to static content (no server-side rendering), but very cheap, very fast, and zero server management.

ADR-006: Reordering Validation Strategy

Decision: Dynamic re-evaluation of items, inventory, and pricing at checkout instead of copying old invoices.

Context: Reordering clones a past checkout. Over months, menu prices, taxes, and item availability change. Simply copying old values could sell retired products or under-charge orders.

Trade-offs: Adds minor latency, but prevents out-of-stock complaints, guarantees pricing accuracy, and ensures tax compliance.

ADR-007: Favorites Custom Configurations Storage

Decision: Serialize custom pizza options as JSON strings inside a NVARCHAR(MAX) column in ordering.Favorites, rather than using child join-tables.

Context: Customers customize toppings, sizes, and crusts. Joining Favorites with dynamic toppings, add-ons, and catalog structures creates heavy SQL queries.

Trade-offs: Prevents advanced database queries on specific toppings, but optimizes catalog load times, reduces relational DB overhead, and simplifies REST API payloads.
